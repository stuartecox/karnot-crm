Design Philosophy

Five Core Design Decisions

Before the architecture: the strategic choices that shape the whole system

🤝

Decision 1: Work WITH Kraken, Not Against It

Building a full VPP dispatch platform like Kraken would take 3+ years and tens of millions. Octopus Energy already built it. Karnot's advantage is the asset layer, not the grid dispatch layer.

The Plan: Build the best asset control and data platform in SE Asia. Feed that data to Kraken via their published FLEX API. Kraken does the grid dispatch. Karnot earns from the assets.

🏗️

Decision 2: Modbus TCP/IP as the Field Bus

Every heat pump, inverter, battery, fan coil, and energy meter made in the last 20 years supports Modbus. BACnet/IP is added for newer HVAC controllers. This ensures maximum compatibility with minimum custom integration work.

Rule: Modbus TCP/IP over Ethernet first. RS485/Modbus RTU via gateway for older equipment. BACnet/IP for premium HVAC. SG-Ready 4-mode relay as the universal fallback for any heat pump.

📡

Decision 3: iSAVE Gateway is the SCADA Node

Every site has one iSAVE Gateway Controller — a Linux-based DIN-rail computer that runs local control logic, collects all field data, executes DR commands, and buffers 30 days of data locally. The site works even when the internet is down.

The iSAVE Gateway is the brain on-site. Utterberry Base Station plugs into it. GUDE relay boxes connect via LAN. All field devices connect to it via RS485 or Ethernet.

🔗

Decision 4: Rubidex is the Data Notary, Not the Database

Karnot owns its own time-series database (TimescaleDB). Rubidex provides the immutable proof layer — every 15-minute energy hash is committed to their blockchain. Utilities and carbon registries get cryptographic proof the data is genuine.

Rubidex = audit trail + carbon credit verification. Karnot = operational platform + fleet management. These are two different things and both are needed.

🇵🇭

Decision 5: Design for Philippines First, Scale to the World

Every specification is made for the Philippines reality: 220V 60Hz grid, frequent brownouts (LTE-M primary + 4-hour battery backup), tropical climate (45°C, 95% RH, IP65 minimum), and commercial cooling (not heating) as the primary application. The European/UK variant adds Matter certification and heating optimisation later.

Protocols

Communication Protocols — What Talks to What

Every connection in the system, with the exact protocol and why

🔬 Utterberry Sensors → iSAVE Gateway

Sub-GHz Wireless Mesh Utterberry Proprietary   Utterberry motes form a self-healing mesh and relay data to the Utterberry Base Station. The Base Station connects via Ethernet to the iSAVE Gateway LAN port. Action needed: Ask Heba whether Base Station outputs (a) local MQTT on LAN, or (b) only to utterberry.io cloud. If local MQTT: iSAVE Gateway subscribes directly. If cloud-only: iSAVE Cloud polls utterberry.io REST API. Either works — local is preferred for brownout resilience.

🔥 Karnot iHEAT Heat Pump → iSAVE Gateway

RS485 Modbus RTU SG-Ready Relay (backup)   Primary: RS485 Modbus RTU at 9600 baud. iSAVE reads 30+ registers: compressor status, setpoint, inlet/outlet temps, flow rate, power draw, fault codes. Writes: setpoint, mode (heat/cool/DHW/off), fan speed. Backup: 4 SG-Ready relay outputs on Gateway (NC/NO contacts) map to all 4 SG-Ready operating modes regardless of Modbus status. This ensures DR control even if Modbus fails.

☀️ Solar Inverter → iSAVE Gateway

SunSpec Modbus TCP/IP Modbus RTU (fallback)   SunSpec is an open standard. Sungrow, Huawei, Fronius, SMA, SolarEdge all implement it. iSAVE reads: AC power output (W), daily/total generation (kWh), DC voltage/current per string, grid frequency, inverter temp, fault status. Over Ethernet via Modbus TCP — no adapter needed for modern inverters.

🔋 Electric Battery (BESS) → iSAVE Gateway

Modbus TCP/IP RS485 Modbus RTU CAN 2.0B (CATL)   BYD/Pylontech: RS485 Modbus RTU. iSAVE reads: SoC (%), SoH (%), cell voltages, pack temperature, charge/discharge current, available capacity. Writes: charge power limit, discharge power limit, operating mode. CAN bus required for CATL large-format systems — CAN-to-Modbus gateway adds $40.

🔌 GUDE DR Relay Controllers → iSAVE Gateway

HTTP REST (LAN) SNMP (LAN)   GUDE units connect via standard Ethernet on the local LAN. iSAVE Gateway sends HTTP GET/POST commands to switch relays: GET http://192.168.1.50/statusjsn.js?components=.... Response time <100ms. GUDE 2314-1 also reports real-time kWh per output channel — fed back to iSAVE for DR verification logging.

🌬️ Fan Coil Units → iSAVE Gateway

RS485 Modbus RTU BACnet MS/TP BACnet/IP   Karnot iZONE: RS485 Modbus bus, up to 32 units per bus. BACnet controllers: BACnet/IP over Ethernet or MS/TP over RS485. iSAVE reads/writes: zone setpoint, actual temperature, fan speed (1/2/3/auto), heating/cooling valve position, occupancy mode, filter alarm.

⚡ Energy Meters → iSAVE Gateway

RS485 Modbus RTU   Eastron SDM630 and Carlo Gavazzi EM340 both use RS485 Modbus RTU at 9600 baud. iSAVE reads: kW (per phase), kVAr, kWh import/export, power factor, frequency, voltage, current. Polled every 10 seconds. All meter readings hashed to Rubidex every 15 minutes.

🔗 iSAVE Gateway → Rubidex Blockchain

HTTPS REST API Rubidex SDK   Every 15 minutes, iSAVE Gateway bundles: timestamp, site ID, energy readings (kWh import/export, generation, battery SoC), active DR events, Utterberry sensor snapshots. This bundle is SHA-256 hashed locally, signed with the device's hardware key (ATECC608B), then submitted to Rubidex via their API. Rubidex commits the hash to their permissioned blockchain. If LTE is down, hashes are queued locally and submitted in batch when connectivity resumes — no data is ever lost.

☁️ iSAVE Gateway → Karnot Cloud (MQTT)

MQTT over TLS 1.3 LTE-M primary Wi-Fi secondary   All sensor data published to EMQX broker in Karnot Cloud. Topic structure: karnot/{site_id}/{device_type}/{device_id}/{datapoint}. QoS 1 (at-least-once delivery) for energy data. QoS 2 (exactly-once) for DR commands. Heartbeat every 30 seconds. Reconnect with exponential backoff. 30-day local buffer ensures no data loss during outages.

☁️ Karnot Cloud → Octopus Kraken FLEX API

Kraken FLEX REST API HTTPS   Karnot Control Centre registers the fleet with Octopus Kraken as a FLEX asset aggregator. Every 30 minutes, Karnot pushes: (a) available DR capacity (kW) per site, (b) forecast availability for next 24 hours, (c) response time per site. Kraken dispatches DR events back to Karnot via webhook. Karnot translates Kraken DR signals into OpenADR 2.0 events and pushes to the relevant site Gateway. This arrangement means Kraken does the grid dispatch optimisation and Karnot earns DR payments from Kraken.

SG-Ready Standard

SG-Ready Implementation in iSAVE

iSAVE implements all 4 SG-Ready modes as required by the German BWP standard — used globally as the de-facto smart grid heat pump interface

🚫

Mode 1 — Utility Block

Heat pump is locked off. iSAVE closes relay contact 1. Duration: max 2 hours, up to 3× per day. Thermal battery continues to supply heating/cooling. Electric backup element can be enabled. Triggered by: utility DR signal or frequency event.

Relay 1: CLOSED   Relay 2: OPEN
✅

Mode 2 — Normal Operation

Standard efficient operation. Both relay contacts open. iSAVE optimises scheduling, pre-charges thermal battery before expected block period, and adjusts setpoints for off-peak tariff. This is the default running state.

Relay 1: OPEN   Relay 2: OPEN
☀️

Mode 3 — Solar Surplus Boost

PV generation surplus detected. Relay contact 2 closed. Heat pump increases to intensified heating/DHW mode — using excess solar to charge thermal battery. Recommended (not mandatory) start command. iSAVE calculates surplus threshold before triggering.

Relay 1: OPEN   Relay 2: CLOSED
⚡

Mode 4 — Active Command On

Definitive command to actively switch heat pump on, optionally with temperature increase in storage. Triggered by: cheap off-peak tariff, grid oversupply event, pre-DR pre-conditioning. Both relay contacts closed. iSAVE configures temperature boost level per site.

Relay 1: CLOSED   Relay 2: CLOSED

🔌 iSAVE SG-Ready Wiring (Universal)

The iSAVE Gateway provides 4 potential-free relay outputs (terminals SG1+/SG1−, SG2+/SG2−). These wire directly to the heat pump controller's SG-Ready or EVU (Energieversorger) terminals — a simple 2-wire connection per contact. Works with Daikin, Mitsubishi, Panasonic, Vaillant, Viessmann, Worcester Bosch, and any BWP-compliant heat pump regardless of manufacturer or age. No programming required on the heat pump side — it just sees the contact states and responds per its factory SG-Ready logic.

Rubidex + Carbon

Rubidex Blockchain & Carbon Credit Pipeline

How every kWh saved becomes a verified, tradeable carbon credit via Rubidex and Gold Standard

📊 Step 1 — Establish Baseline (at Installation)

  • iSAVE Gateway records 30-day pre-installation energy baseline from utility meter
  • Baseline model committed to Rubidex blockchain (tamper-proof)
  • Weather-normalised baseline using local temperature data
  • Approved by Karnot engineer and committed with timestamp + GPS coordinates
  • Rubidex issues a unique Device Credential NFT for this installation

⚡ Step 2 — Continuous Measurement (Every 15 min)

  • iSAVE reads energy meter: actual kWh consumed this period
  • Calculates: Baseline kWh − Actual kWh = Saved kWh
  • Applies: Philippine Grid Emission Factor (kgCO₂/kWh from DOE)
  • Calculates: Avoided CO₂ in tonnes for this period
  • Bundle signed with device private key → submitted to Rubidex API
  • Rubidex commits hash to permissioned blockchain

🏦 Step 3 — Monthly Credit Batch

  • At month end, Karnot aggregates all Rubidex-verified avoided CO₂ for each site
  • Fleet total submitted to Gold Standard registry API as a batch claim
  • Gold Standard reviews blockchain proof — no site visit required (blockchain is the audit)
  • Credits issued: 1 VER (Verified Emission Reduction) = 1 tonne CO₂ avoided
  • Credits appear in Karnot's Gold Standard registry account

💰 Step 4 — Revenue Distribution (Smart Contract)

  • Credits sold on voluntary carbon market (Xpansiv CBL, AirCarbon Exchange, or direct to corporates)
  • Philippines voluntary carbon price: ~$8–15/tonne CO₂ (growing)
  • Rubidex smart contract auto-distributes: 70% to site owner, 15% to Karnot, 15% to operating costs
  • Distribution ratios configurable per contract
  • Full audit trail from sensor reading → credit sale on Rubidex ledger

🇵🇭 Philippines Carbon Market Specifics

The Philippines Climate Change Commission (CCC) launched the Philippine Carbon Market under RA 11285 (Energy Efficiency Act). Karnot iSAVE data — with Rubidex blockchain verification — is ideally positioned to generate credits under: (a) Gold Standard's Efficient Cookstoves/HVAC methodology, (b) CDM/SD-VISta methodology for demand-side efficiency projects, (c) Future Philippine voluntary carbon exchange (in development under DOE). Grid emission factor for Luzon: ~0.6 kgCO₂/kWh (2024, DOE Philippines).

Karnot vs Kraken

Karnot Control Centre — What to Build vs What to Buy

Stuart's instinct to work WITH Kraken rather than copy it is correct. Here's the exact division of responsibility.

🏗️ What Karnot Builds (iSAVE + Control Centre)

  • iSAVE Gateway: The on-site controller. This is Karnot's hardware product.
  • Asset data collection: Every reading from every device, every 10 seconds.
  • Local AI control: Solar optimisation, battery dispatch, thermal pre-conditioning.
  • Fleet dashboard: Karnot engineers see all sites, all assets, all alarms.
  • Carbon credit pipeline: Rubidex-verified MRV → Gold Standard submission.
  • Predictive maintenance: AI-flagged anomalies → service tickets.
  • Utility portal: Philippine DUs see their enrolled DR fleet capacity.
  • OpenADR VTN: Issue DR commands to all Gateways from Control Centre.

🐙 What Kraken Does (Karnot Feeds, Kraken Dispatches)

  • Grid signal intelligence: Balancing Mechanism prices, grid frequency, carbon intensity forecasts.
  • Optimal dispatch decisions: When to curtail which assets, for how long, at what price.
  • Settlement & payment: Kraken handles the financial settlement with National Grid / System Operator.
  • Aggregation at scale: Kraken aggregates Karnot's fleet alongside thousands of other assets.
  • Market access: Kraken is registered with every grid operator. Karnot doesn't need to be.

📡 Integration: Karnot registers fleet assets via Kraken FLEX API. Karnot pushes half-hourly asset availability. Kraken sends back dispatch signals. Karnot translates to OpenADR for Gateways. DR payment flows: Kraken → Karnot → site owner via smart contract.

🇵🇭 Philippines-Specific: Meralco RDSP Instead of Kraken

In the Philippines, the equivalent is Meralco's RDSP (Retail DR Service Provider) programme and the IEMOP (Independent Electricity Market Operator of the Philippines) ancillary services market. Karnot registers as an RDSP aggregator with Meralco. The Karnot Control Centre issues DR signals to enrolled sites. Load shed is verified via Rubidex blockchain records. Meralco pays Karnot per kW of verified load reduction. Karnot distributes payments to site owners via smart contract. Same model as Kraken — just the Philippine grid operator instead.
