OpenADR 2.0b VEN — DR in <100ms

Gateway registers as Virtual End Node (VEN) with Meralco/VECO OpenADR VTN. Receives oadrDistributeEvent. Executes SG-Ready Mode 1 (block) via Modbus within <100ms. Reports verified curtailment via oadrCreatedEvent for payment.

Data Trust Layer

Rubidex:
The Immutable Data Notary

Because we use our own Karnot sensors, we need absolute proof the data is real. Rubidex blockchain is how Meralco, VECO, and Gold Standard will accept our DR and carbon claims — automatically, without manual audit.

Trust Chain: Device → Blockchain → Payment

1.ATECC608B — hardware-signed at source. Private key physically irretrievable from silicon. Every packet provably from that device.
2.Every 15 minutes — Gateway builds SHA-256 Merkle root of signed packets. Commits to Rubidex permissioned blockchain. Immutable timestamp.
3.VECO / Meralco — queries Rubidex chain. Sees ECDSA-signed curtailment + blockchain timestamp. Automated payment. No disputes possible.
4.Gold Standard registry — accepts Rubidex as MRV evidence. Carbon credits issued with zero manual audit. Full pipeline autonomous.

About Rubidex (formerly Penteon)

Permissioned blockchain for industrial IoT data notarisation. "Data Fabric" underpins BMSIntel (buildings) and GridLock (grid). iSAVE provides the data layer at the intersection of both products.

Six Patent Claims:
The iSAVE IP Portfolio

Claim 1 — Device
iSAVE Bridge: PoE-Powered Modbus-to-MQTT Gateway with Hardware Crypto
A field device comprising: PoE 802.3af power; RS485 Modbus RTU interface to SG-Ready heat pump; ATECC608B hardware crypto processor; Ethernet MQTT output with ECDSA-signed telemetry; operable to control heat pump operating modes via Modbus SG-Ready commands.
Claim 2 — Method
Native SG-Ready Demand Response Execution via Modbus
A method: receiving OpenADR 2.0b event signal; pre-charging building thermal mass via SG-Ready Mode 4 (Modbus command); switching to Mode 1 (curtailment) on DR trigger; maintaining comfort from stored thermal energy ≥2 hours; transmitting Rubidex-verified curtailment report for utility payment.
Claim 3 — Method
Blockchain-Verified Demand Response Metering (Native Sensor)
A metering method: reading native heat pump energy data via Modbus; cryptographically signing at ATECC608B; aggregating signed data at Gateway; committing SHA-256 hash every 15 minutes to permissioned blockchain; providing blockchain transaction as tamper-proof curtailment evidence for payment.
Claim 4 — System
Legacy Retrofit Kit: Third-Party Equipment DR Integration
A retrofit system comprising: Utterberry sub-GHz sensor motes (temperature, vibration, pressure); GUDE IP relay controller for non-SG-Ready load switching; iSAVE Bridge (Claim 1) for data aggregation and signing; same Rubidex blockchain notarisation pipeline as native deployments.
Claim 5 — Method
Autonomous Carbon Credit MRV from Native IoT Data
An MRV method: continuous energy measurement via native Modbus registers; baseline comparison against pre-deployment consumption profile; blockchain notarisation of verified savings every 15 minutes; automated submission to carbon registry (Gold Standard / Verra VCS); credit issuance without manual audit.
Claim 6 — Complete System
Integrated Smart Energy Control, DR, and VPP Platform
An integrated system combining Claims 1–5 with fleet management across plural sites; OpenADR 2.0b VTN relay to utility DR programmes; aggregated DR capacity dispatch as Virtual Power Plant; carbon credit portfolio management; API compatibility with third-party VPP platforms (Octopus Kraken FLEX API schema).
